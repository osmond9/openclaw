# BST 系统完整架构设计文档 V2.0

> **Battery Safety & Technology Platform**
> 以 Apple + Linear.app 风格为主，融合 Notion、Airtable 设计优势
> 面向研发部门的全栈智能管理平台

---

## 🌐 一、系统全局定义

### 1.1 系统定位

```
BST Platform = 
  研发项目管理 + 安全评审流程 + 绩效考核 + 知识数据库 + AI智能助手
  
核心价值：
  让研发工程师专注于技术本身
  让管理者实时掌握团队状态
  让数据自动流转，减少重复录入
  让AI辅助决策，提升管理效率
```

## 📌 十九、关键设计决策备忘

```
┌──────────────────────────────────────────────────────────────────┐
│  关键设计决策                                                     │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Q1: 安全评审数据是否需要防篡改？                                  │
│  A:  是。所有评审记录归档后设为只读，修改需要创建新版本，           │
│      历史版本永久保留，操作全部记录审计日志。                       │
│                                                                  │
│  Q2: AI 模式下数据安全如何保障？                                   │
│  A:  优先考虑私有化部署 LLM，数据不出内网。                        │
│      若使用云端 API，需对敏感字段（人名/产品型号）脱敏后再传输。    │
│      AI 权限与用户权限绑定，AI 看不到用户看不到的数据。            │
│                                                                  │
│  Q3: 绩效数据的保密性如何处理？                                    │
│  A:  员工只能看到自己的绩效分数。                                  │
│      负责人能看到直接下属的分数（加密传输）。                      │
│      部门排名仅展示相对名次，不直接展示他人具体分数。              │
│                                                                  │
│  Q4: 危害分析标准库更新后，历史评审如何处理？                      │
```
